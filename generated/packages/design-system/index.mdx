## Overview & Design Philosophy

The `@cityos/design-system` package is a comprehensive design system designed to support the Dakkah CityOS monorepo. It provides a set of reusable components, hooks, and theming capabilities that ensure consistency across various platforms and applications within the ecosystem.

## Design Tokens, Component Primitives, and Theming Architecture

### Design Tokens
Design tokens are centralized values that define the visual language of the design system. They include colors, typography, spacing, and other styling properties. These tokens are defined in the `@cityos/design-tokens` package and are used across all components to maintain consistency.

### Component Primitives
The design system includes a set of low-level component primitives that serve as building blocks for more complex components. These primitives ensure a consistent look and feel across the platform and facilitate rapid development.

### Theming Architecture
Theming is managed through a theming architecture that allows for customization and extension of the default theme. The `@cityos/design-system` package provides a set of default themes, which can be overridden or extended by individual applications to meet specific branding requirements.

## Accessibility Standards (WCAG) and Inclusive Design Patterns

The design system adheres to WCAG 2.1 accessibility standards to ensure that all users, including those with disabilities, can access and interact with the platform effectively. Key inclusive design patterns include:

- **Keyboard Navigation**: All interactive components are fully keyboard navigable.
- **Screen Reader Support**: Components are designed to be accessible to screen readers, ensuring that all content is readable by assistive technologies.
- **Color Contrast**: The design system enforces sufficient color contrast ratios to meet WCAG guidelines.
- **Aria Attributes**: Appropriate ARIA attributes are used to enhance the accessibility of interactive components.

## Cross-Platform Strategy (Web vs Native vs Embedded)

The `@cityos/design-system` package is designed with a cross-platform strategy in mind, supporting web, native, and embedded platforms. This is achieved through:

- **Universal UI Components**: The design system includes universal UI components that can be rendered on both web and native platforms using the `@cityos/universal-ui` library.
- **Platform-Specific Implementations**: For platform-specific features or optimizations, the design system provides hooks and utilities that allow developers to implement platform-specific logic.

## Component Composition Patterns and Slot Architecture

Components are composed of smaller, reusable components following a modular architecture. This approach allows for easy customization and extension of individual components while maintaining consistency across the design system.

## Architecture

The `@cityos/design-system` package is a comprehensive design system built using Node.js, designed to provide a consistent look and feel across various applications within the CityOS ecosystem. This document outlines the architecture of the package, focusing on token hierarchy, component implementation patterns, cross-renderer support, build optimization, and integration with design tools.

## Token Hierarchy

The design tokens in `@cityos/design-system` are organized into a hierarchical structure to ensure consistency and scalability:

1. **Primitive Tokens**: These are the foundational values that define basic properties such as colors, typography, spacing, and sizing.
2. **Semantic Tokens**: Built on top of primitives, semantic tokens represent higher-level design concepts like button styles or card layouts.
3. **Component Tokens**: Specific to individual components, these tokens provide detailed styling for each component type.
4. **Theme Tokens**: These tokens allow for theming by overriding the default values with custom themes.

## Component Implementation Patterns

The package employs several patterns to ensure flexible and reusable component implementations:

1. **Compound Components**: A compound component pattern is used where a parent component manages the state and behavior of its child components, allowing for complex interactions.
2. **Controlled Components**: These components manage their own internal state but allow external control through props, enhancing reusability and predictability.
3. **Headless Components**: Designed to provide functionality without any UI, headless components are useful for creating highly customizable interfaces.

## Cross-Renderer Architecture

To support both web and native platforms, the design system uses a cross-renderer architecture:

1. **Web Component Variants**: Components are designed using React or other web frameworks, ensuring they work seamlessly in browser environments.
2. **Native Component Variants**: For mobile applications, native components are implemented using React Native or similar frameworks, maintaining platform-specific performance and behavior.

## Build and Tree-Shaking Optimization

The package is optimized for build size and performance through:

1. **Lazy Loading**: Components are lazily loaded to reduce initial bundle size, improving load times.
2. **Tree Shaking**: Unused code is automatically removed during the build process, ensuring that only necessary modules are included in the final bundle.

## Design Tool Integration

The design system integrates with popular design tools to facilitate collaboration and development:

1. **Figma**: The design tokens and components are exported to Figma for visual design and prototyping.
2. **Storybook**: A Storybook setup is provided to document and showcase all components, allowing developers to explore and test them in isolation.
